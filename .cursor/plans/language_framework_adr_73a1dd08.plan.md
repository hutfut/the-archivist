# Language and Framework ADR

A single ADR file at `[.docs/adr/0001-use-python-fastapi-react.md](.docs/adr/0001-use-python-fastapi-react.md)`, plus an index update in `[.docs/adr/README.md](.docs/adr/README.md)`.

## Recommended decision: Python + FastAPI (backend), React (frontend), LangChain/LangGraph (orchestration)

### Key reasoning

- **LangChain/LangGraph is Python-first**: The prompt explicitly requires LangChain/LangGraph. The Python SDK is the canonical implementation -- richer API surface, better docs, larger community, faster feature releases. The JS port lags and has less coverage.
- **Document processing ecosystem**: Python offers mature, battle-tested libraries for parsing PDFs, DOCX, and other formats (`pypdf`, `python-docx`, `unstructured`). Node equivalents are less robust.
- **FastAPI strengths**: Automatic OpenAPI/Swagger docs, Pydantic request/response validation, async-native, dependency injection -- all production-quality with minimal boilerplate.
- **React is the specified UI**: No decision needed; the prompt mandates it.
- **Prompt explicitly allows Python**: "Node.js or Python for server-side components" -- Python is a first-class option.

### Trade-offs accepted

- Two languages (Python + TypeScript) means two dependency management systems and no shared types between frontend and backend.
- Developers must be comfortable in both ecosystems.
- Mitigated by: clear API contracts (OpenAPI spec auto-generated by FastAPI), separate processes regardless of backend language choice.
